# Container Isolation

In these steps we will run some commands on the ```host``` (the Linux Virtual machine hosting a Container Engine)
and other commands within Containers of a Pod.

This will demonstrate that the Pod Containers have isolated:
- Root filesystem
- Processes
- Networking

A process running on the ```host``` system normally operates in the ```global``` Filesystem, Process and Networking ```Namespaces(kernel)```.

A process running inside a Container operates in it's own ```private``` Filesystem, Process and Networking ```Namespaces(kernel)```.

## 1. Creating a Pod

We can create a Pod using the ```kubectl run``` command, or from a YAML (or JSON) manifest.

<details>
  <summary>Creating a new Pod (no shell)</summary>
  ```kubectl run pod1 --image mjbright/k8s-demo:1
</details>

**Note**: To check that the Pod is running, we can use the ```kubectl get pod``` command producing output such as
```
NAME                       READY   STATUS    RESTARTS          AGE
pod1                       1/1     Running   0                 5s
```

## 2. Obtaining a shell in the Pod

### 2.1 No shell

We can obtain a shell in the Pod using the ```kubectl exec``` command, specifying the path to the shell.

But in this example this is not possible because the chosen image for the Pod Container does not include basic Linux commands:

**Note:** the following produces an error:

<details>
  <summary>kubectl exec to pod1</summary>
  ```kubectl exec -it pod1 -- /bin/sh```
</details>

The error message will be similar to:
```
error: Internal error occurred: Internal error occurred: error executing command in container: failed to exec in container: failed to start exec "d0f30824c82fe771b8f0a261513ed536f8b32a952e3f5c38737f9e57777676fc": OCI runtime exec failed: exec failed: unable to start container process: exec: "sh": executable file not found in $PATH
```

**Note:** This demonstrates already that the Container in the Pod has it's own **isolated filesystem**, as the usual shell /bin/sh is not present.

```On the host```: You can check on your host system that the shell /bin/sh does exist there - but not in the Container.

### 2.2 Pod2 - with a shell

The files in a Container are provided by the Container Image used.

Let's create a second Pod, using a more complete Container Image:
- the image includes the same application, but based on a very small ```Alpine Linux``` base:

<details>
  <summary>Creating a new Pod (with a shell)</summary>
  ```kubectl run pod2 --image mjbright/k8s-demo:alpine1
</details>

**Note**: Check that the Pod ```pod2``` is running, as well as the previous Pod ```pod1``` using the ```kubectl get pod``` command producing output such as
```
NAME                       READY   STATUS    RESTARTS          AGE
pod1                       1/1     Running   0               2m18s
pod2                       1/1     Running   0                 7s
```

#### Exec into pod2:

<details>
  <summary>kubectl exec to pod2</summary>
  ```kubectl exec -it pod2 -- sh```
</details>

This time we have a shell.

Running the command ```ps aux``` in the shell will show all processes, you should see output similar to
```
/app # ps aux
PID   USER     TIME  COMMAND
    1 root      0:00 /app/demo-binary -l 80 -L 0 -R 0
    9 root      0:00 /bin/sh
   16 root      0:00 ps aux
/app #
```

```On the host```: Compare this with running ```ps aux``` on the host

# Summary

We have demonstrated that a shell within a Pod Container has a completely different view of
- Files
- Processes
- Networking

This demonstrates the utility of Containers.

This **isolation** allows to run ```many instances``` of ```many applications``` on the same machine without conflicts
- between files, e.g. conflicting libraries
- between processes
- between ip addresses
